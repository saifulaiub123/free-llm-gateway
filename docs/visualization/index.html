Free LLM Gateway

A self-hosted, free-first, multi-user LLM routing gateway — one OpenAI-compatible /v1 endpoint that aggregates many providers' free tiers and routes every request through a configurable, metric-driven fallback chain.

NestJS server Drizzle ORM · PG + SQLite SvelteKit 5 client OpenAI wire-compatible 11 providers 5 routing strategies

1 The Problem We're Solving

Today, LLM access (in ScraperQ and most apps) is wired with hard-coded fallback, no cost awareness, and a single shared config. The gateway replaces all of that.

Pain #1 · Hard-coded fallback

Provider order lives in appsettings.json. Adding a provider or model means an edit + redeploy.

Pain #2 · No free/paid awareness

The system can't prefer free capacity and spill to paid only when free is exhausted. Cost control is manual.

Pain #3 · No live health

On 429/5xx there's no shared cooldown ledger — the next call hits the same exhausted key.

Pain #4 · No model metadata

No stored intelligence, speed, cost, context window, or capability data to route intelligently.

Pain #5 · Single tenant config

Keys live in server config — no per-user ownership, no isolation, no bring-your-own-key.

Pain #6 · Reference doesn't fit

freellmapi is single-user, Node/SQLite-only, and can't back a multi-tenant platform.

2 The Solution at a Glance

Clients change only their base_url + bearer key and pass model: "auto". The gateway picks the cheapest healthy free model and falls back intelligently.

🆓 Free-first

Always try free models first; only spill to paid when free capacity is exhausted, and report cost saved.

🔌 One endpoint

Aggregate many providers behind a single OpenAI-compatible /v1. Clients just swap base_url.

👥 Multi-user

Login/register, per-user encrypted keys, per-user catalog & routing, strict user_id isolation.

🔎 Auto-discovery

Adding a key fetches the provider's models and classifies which are free — one adapter per provider.

🧭 5 strategies

Manual, Free-First, Balanced, Fastest, Smart — each reorders the chain from live metrics.

📊 Live metrics

Availability, rate-limit headroom, stability, intelligence, speed, cost feed every decision.

🔁 Smart fallback

Skip disabled / unhealthy / cooled-down / over-cap candidates; up to 20 attempts.

📈 Analytics

Per-request logging, success rate, latency, tokens, and the headline cost-saved metric.

🤝 Contract-compatible

Honors ScraperQ's existing contract: /v1, model:"auto", X-Routed-Via, X-Fallback-Attempts.

4 A Request's Journey

What happens when a client POSTs to /v1/chat/completions with model: "auto".

sequenceDiagram autonumber participant C as Client (ScraperQ) participant G as LlmApiTokenGuard participant R as RoutingService participant F as ChainFilter participant O as ChainOrderer participant X as FallbackExecutor participant A as Provider Adapter participant U as Upstream Provider participant L as Request Log C->>G: POST /v1/chat/completions (Bearer sqr-llm-…) G->>G: hash token → lookup api_tokens (reject JWTs) G->>R: buildChain(user, "auto", strategy, capabilities) R->>F: load candidates → filter F-->>R: drop disabled / unhealthy / cooldown / over-cap / incompatible R->>O: order by selected strategy O-->>R: ordered fallback chain R->>X: execute(chain) loop until success or MAX_FALLBACK_ATTEMPTS (20) X->>A: chatCompletion(candidate, key) A->>U: upstream call alt success U-->>A: 200 OK A-->>X: response X->>L: record usage + runtime stats + cost saved else 429 / 5xx / timeout U-->>A: error X->>X: place key on cooldown, attempts++ end end X-->>C: OpenAI-shaped response
X-Routed-Via + X-Fallback-Attempts

🔎 Filter (SRP)

ChainFilter removes candidates with no healthy key, capability mismatch, hard cap exceeded, or active cooldown.

🧭 Order (SRP)

ChainOrderer applies the resolved IRoutingStrategy + its config to sort the survivors.

🔁 Execute (SRP)

FallbackExecutor walks the chain, cools down failures, records outcomes, stops at 20 attempts.

📨 Response headers (the contract)

X-Routed-Via: groq/llama-3.3-70b  ·  X-Fallback-Attempts: 2 — the exact headers ScraperQ's FreeLlmApi client already parses, so the Backend works unchanged.

6 Data Model

Every table composes shared base columns. User-facing entities add audit fields + soft delete; high-volume ledgers are append-only.

🧬 Base column sets (DRY)

  • baseColumns — id, createdAt on every table
  • baseEntityColumns — adds createdBy, modifiedBy, modifiedAt, isActive, isDeleted on user-facing entities
  • Append-only ledgers (request_logs, rate_limit_counters, cooldowns) use only base columns — hard delete / TTL
  • FKs are real + indexed; createdBy/modifiedBy → users.id ON DELETE SET NULL

🗄️ Pluggable database layer

  • Each engine is a self-contained ProviderModule under providers/<name>/
  • Provides: prefix-applying table creator, ColumnKit, audit helper, connect, migrator
  • One providerRegistry resolves the active module via getActiveProvider()
  • Same repository body runs on PostgreSQL and SQLite (both async)
  • Configurable DB_SCHEMA + DB_TABLE_PREFIX

🔐 Provider keys

AES-256-GCM encrypted at rest; decrypted only in memory right before an upstream call.

#️⃣ API tokens

SHA-256 hash stored; plaintext shown exactly once at creation.

🔑 Passwords

Argon2id hashing.

🔄 Refresh tokens

Hashed, rotated on use, revocable, replay-detected by family.

👑 First-user admin

First registration → Admin (zero-config); never lockable-out.

🚪 Registration toggle

Admins enable/disable self-registration (403 when closed).

🛡️ RolesGuard

/api/v1/admin/* requires the Admin role.

🙈 No secret leaks

ENCRYPTION_KEY & keys never logged or returned.

9 Why It's Worth Building

The payoff: lower cost, higher reliability, true multi-tenancy, and a contributor-friendly codebase.

Cost

Maximized free-tier usage. Always try free first; spill to paid only when exhausted, with a measurable “cost saved” headline metric.

Reliability

Self-healing fallback. Live cooldowns + rate-limit headroom mean a rate-limited key is skipped, not retried blindly.

Flexibility

Zero-redeploy extensibility. New providers/strategies/DBs are additive — the router core never changes.

Isolation

True multi-tenancy. Per-user encrypted keys, per-user catalog & routing, user_id-scoped everywhere.

Compatibility

Drop-in for OpenAI clients. Swap base_url + key; ScraperQ Backend runs unchanged.

Openness

Contributor-friendly. TypeScript, SOLID/SRP, small units, mandatory tests & docs per task.

🎯 The 10 goals, condensed

  • Maximize free-tier usage → minimize cost (surface “cost saved”)
  • Aggregate many providers behind one OpenAI-compatible endpoint
  • Multi-user from day one with strict data isolation
  • Auto-discover free models per provider (provider-specific logic)
  • Support user-defined custom models + per-model enable/disable
  • Offer Manual + Balanced + Smart + Fastest + Free-First strategies
  • Save model order separately per routing strategy
  • Track rate-limit usage & cooldowns per (user, provider, model, key)
  • Honor ScraperQ's existing HTTP contract
  • Be open source & maintainable (SOLID, SRP, SoC)
